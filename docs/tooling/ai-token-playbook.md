# AI 协作低 Token 操作手册

这份手册用于降低 `black-tonny-frontend` 仓库里的 AI 协作会话 token 消耗。

它是执行辅助，不替代 [../../AGENTS.md](../../AGENTS.md) 里的优先级、读顺序和仓库规则。

## 适用场景

- Codex / ChatGPT / Claude / Gemini 等 AI 编码协作
- repo-local 文档导航、问题排查、方案分析
- repo-local tooling、temp skill、跨仓库文档跳转

不适用场景：

- 页面业务口径定义
- dashboard 指标语义
- 运行时登录或接口契约本身的业务决定

这些内容仍以对应业务文档和标准文档为准。

## 最小读法流程

1. 先用 [../../AGENTS.md](../../AGENTS.md) 里的 task routing 判断任务类型。
2. 默认只打开 1 到 2 个最相关入口文档，不先扫全量入口。
3. 先读相关章节或相关片段，再决定是否需要整篇阅读。
4. 只有当任务触达契约、边界、数据流、source-of-truth 规则或 maintainer 决策时，才继续扩读深层文档。

## 搜索与阅读默认值

- 搜索优先限定在最可能相关的目录和关键字，不默认做 workspace 级宽搜。
- 找前端入口优先按 route、view、layout、component、API 模块名缩小范围。
- 阅读大文件时优先先找标题、段落或关键模块，再按片段读取。
- 工具输出只保留能支撑当前判断的最小证据，不把长搜索结果整段带进后续多轮。
- 如果已经确认某条事实，后续优先引用摘要，不重复贴原文。

## 跨仓库扩读建议

- 如果只是前端页面、布局、组件、mock login、前端工程标准问题，先停留在 frontend 仓。
- 只有当任务明确涉及 backend API 契约、capture / serving、MCP、browser research 或 frontend-backend boundary，才进入 backend 对应文档。
- 跨仓库任务优先先看 [文档总图](../document-map.md)，不要凭记忆直接跳多个深层文档。

## 事实缓存模板

每轮探索后，建议沉淀 3 到 5 行固定摘要：

```text
已确认：
- …

已排除：
- …

下一步：
- …
```

建议规则：

- `已确认` 只保留当前已验证的仓库事实
- `已排除` 只写已经看过且可停止继续追的方向
- `下一步` 只写一个最直接的后续动作

## 何时必须扩读

出现下面任一情况时，不应继续只靠最小读法：

- 要修改公共接口契约、数据源路径或前后端边界
- 要调整登录 / access / route guard 的正式基线
- 要修改 frontend 工程标准、alignment 判断或 maintainer 决策
- 当前证据不足以判断 source-of-truth 文档归属

## 不建议的做法

- 一上来就整篇打开所有入口文档
- 默认执行跨整个 workspace 的宽范围搜索
- 为了进度同步重复转述同一结论
- 在多轮对话里反复粘贴同一段长输出
- 还没确认边界就扩展到无关 area docs

## 简单验收口径

如果这套读法生效，通常会出现下面这些变化：

- 初始探索读取的文件数更少
- 很少出现无边界的大范围搜索
- 同一事实不会被重复读取和重复复述
- 只有真正需要时才进入 backend 深层文档
