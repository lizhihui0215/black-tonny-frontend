# 平台方案说明

状态：`Working doc`

## 1. 方案目标

这套方案的目标，是在不打断现有业务系统使用的前提下，逐步把零售经营相关的数据、页面和分析链路沉淀成可复用、可扩展的产品底座。

它不是一次性把所有功能都做完，而是分阶段推进：

- 先盘清页面与数据路线
- 再确认哪些能力可以稳定准入
- 最后逐步扩大正式可交付范围

## 2. 整体结构

当前平台从业务方案视角可以分成三层：

### 2.1 页面与交互层

负责：

- 页面入口
- 指标卡片
- 页面联动
- 分析展示体验

### 2.2 服务与投影层

负责：

- API 输出
- 指标聚合
- 服务层投影
- 页面消费所需的结构化数据

### 2.3 数据研究与准入层

负责：

- 页面研究
- 接口发现
- 参数语义确认
- 原始数据留痕
- 数据路线准入判断

## 3. 当前方案方法

当前方法不是“发现一个接口就直接上线使用”，而是分阶段推进：

1. 页面研究  
   先确认页面动作、接口、参数和粒度关系。

2. 纯 HTTP 回证  
   用正式 HTTP 请求验证研究结论，而不是长期依赖浏览器。

3. 数据路线准入  
   只有语义、分页、范围切换和证据闭环足够清楚的路线，才进入正式数据底座。

4. 服务层开放  
   只有稳定运行的数据路线，才逐步供上层页面和 API 使用。

这里需要特别注意：

- 这份方案文档负责高层方案表达
- 具体数据路线成熟度、ERP 研究状态和 `capture` 准入状态，仍以后端状态板为准

## 4. 当前重点建设方向

### 已具备

- 当前账号可见范围内的页面覆盖与接口版图盘点
- 经营分析前端页面骨架
- 跨前后端的文档和协作入口
- 面向数据准入的成熟度状态板与路线注册表

### 已在建设中

- 销售和库存等高价值路线的正式准入
- 从原始数据到底层投影的稳定回归机制
- 页面逐步从 sample/fallback 向稳定数据路线迁移

### 规划中

- 会员、储值、流水等更多业务域的深挖与准入
- 更完整的经营动作支持
- 更成熟的交付模板和部署模式

## 5. 适合的实施方式

### 第一阶段：建立页面和数据底座

- 确认系统边界
- 搭起前后端协作框架
- 盘清页面和接口路线
- 建立数据准入机制

### 第二阶段：优先打通高价值业务域

- 销售
- 库存
- 再逐步扩展到会员、储值、流水

### 第三阶段：把稳定能力开放给更多页面和业务分析

- 页面逐步接入更稳定的数据路线
- 服务层投影逐步扩大
- 产品能力从展示走向分析与动作支撑

## 6. 对客户沟通时应如何表述

建议用以下方式表达：

- 已有统一前后端底座和核心分析页面能力
- 已建立数据研究和准入机制
- 正在把高价值业务域逐步纳入正式数据底座
- 当前适合先从重点经营场景落地，而不是一次性承诺覆盖所有历史能力

不建议直接表述成：

- 已完成所有 ERP 接口的正式接入
- 已经拥有最终稳定的数据仓模型
- 所有分析结果都已经完全自动化稳定上线

## 7. 当前交付边界

当前更适合交付的是：

- 核心分析页面与主流程
- 数据底座建设方案
- 高价值路线的研究与准入计划
- 面向后续扩展的协作结构

当前还不应直接承诺的是：

- 所有业务域的正式稳定入库
- 所有页面都完全摆脱研究态数据
- 全角色、全门店、全租户的一次性覆盖
